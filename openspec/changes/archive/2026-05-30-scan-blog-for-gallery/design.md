## Context

当前 `history/generate.py` 脚本直接扫描 `docs/attachments/` 目录结构来提取图片文件。这种方式虽然简单，但存在以下局限：
- 无法利用博客文章中的丰富元数据（标题、日期、标签等）
- 图片展示与内容管理脱节
- 维护两套独立的文件结构（博客内容 + 附件目录）

博客文章存储在 `docs/blog/` 目录下，使用标准的 Markdown 格式。交易周记文章包含"老罗投资周记"关键字，并引用持仓截图：
```markdown
![目前持仓](../../../attachments/2026/03/20260307/1.png)
```

**约束条件**：
- 必须保持现有 HTML 输出格式和模板不变
- 必须兼容现有的文件结构（博客和附件）
- 不能影响博客文章的正常显示
- 生成脚本应独立运行，无需构建流程依赖

## Goals / Non-Goals

**Goals:**
- 重写 `history/generate.py` 的核心扫描逻辑，从扫描附件目录改为解析博客 Markdown 文件
- 实现 Markdown 文件内容解析，提取图片引用和元数据
- 保持日期解析、排序和 HTML 生成功能不变
- 提升图库展示质量，利用博客文章的元数据

**Non-Goals:**
- 不修改博客文章的结构或格式
- 不改变附件目录的组织方式
- 不实现图片上传或管理功能
- 不修改 HTML 模板和样式（除非必要）
- 不添加数据库或动态后端

## Decisions

### 1. Markdown 解析策略

**决策**：使用正则表达式解析 Markdown 文件，不引入完整的 Markdown 解析库。

**理由**：
- **简单高效**：只需要提取图片引用和基本元数据，正则表达式足够
- **零依赖**：不需要额外的 Python 包（如 `markdown` 或 `markdown2`）
- **可控性强**：针对特定格式（`![目前持仓](...)`）的解析更精确

**为什么不使用 Markdown 解析库**：
- 完整的 Markdown 解析库（如 Python-Markdown）会解析整个文档 AST，增加复杂度和内存占用
- 我们只需要提取特定模式的内容，不需要渲染或转换 Markdown
- 正则表达式对于这种简单用例更直接

**实现细节**：
- 使用 `re` 模块匹配图片引用模式：`!\[目前持仓\]\(([^)]+)\)`
- 提取括号内的相对路径
- 支持 `../` 和 `./` 等相对路径前缀

### 2. 文件扫描和过滤

**决策**：递归扫描 `docs/blog/` 目录，使用关键字过滤和文件扩展名过滤。

**实现**：
1. 使用 `pathlib.Path.rglob("*.md")` 递归查找所有 Markdown 文件
2. 读取每个文件内容，检查是否包含"老罗投资周记"关键字
3. 只处理包含关键字的文件

**理由**：
- 递归扫描确保不遗漏任何博客文章
- 关键字过滤确保只处理交易周记文章
- 文件扩展名过滤避免处理非 Markdown 文件

**替代方案考虑**：
- **仅扫描特定日期范围**：不够灵活，无法自动发现新文章
- **基于文件名模式**：博客文件名格式不统一，不可靠

### 3. 路径转换逻辑

**决策**：实现相对路径到公网 URL 的转换函数。

**转换规则**：
- 输入：`../../../attachments/2026/03/20260307/1.png`
- 处理：去除 `../` 前缀，保留 `attachments/` 及后续路径
- 输出：`https://invest.zdyi.com/attachments/2026/03/20260307/1.png`

**实现**：
```python
def convert_to_public_url(relative_path: str) -> str:
    # 去除 ../ 前缀
    clean_path = relative_path.replace('../', '')
    # 确保路径以 attachments/ 开头
    if not clean_path.startswith('attachments/'):
        clean_path = 'attachments/' + clean_path
    return f'https://invest.zdyi.com/{clean_path}'
```

**为什么硬编码域名**：
- 简化实现，避免配置文件
- 当前项目固定部署在 `invest.zdyi.com`
- 未来可以通过命令行参数扩展

### 4. 元数据提取

**决策**：从 Markdown 文件中提取标题和日期，增强图库展示。

**提取策略**：

**标题提取**：
- 优先级 1：从 Markdown frontmatter 的 `title` 字段
- 优先级 2：第一个 `#` 一级标题
- 优先级 3：使用文件名（去除扩展名）

**日期提取**：
- 优先级 1：从图片路径中解析（`.../2026/03/20260307/1.png` → `20260307`）
- 优先级 2：从文件路径中解析（`docs/blog/2026/03/20260307.md` → `20260307`）
- 优先级 3：从 frontmatter 的 `date` 字段

**理由**：
- 多优先级策略确保最大兼容性
- 图片路径是最可靠的日期来源
- 标题信息可用于未来的图库展示增强

### 5. 错误处理和容错

**决策**：实现多层次的错误处理，确保脚本健壮性。

**错误处理策略**：

1. **文件读取错误**：
   - 文件不存在：跳过，记录警告
   - 权限不足：跳过，记录错误
   - 编码问题：尝试 UTF-8，失败则跳过

2. **内容解析错误**：
   - 未找到图片引用：跳过该文件
   - 图片引用格式不匹配：跳过，记录警告
   - 日期解析失败：使用文件名作为标签，放在列表末尾

3. **整体容错**：
   - 任何单个文件的错误不应中断整个生成过程
   - 收集所有警告和错误，在最后统一报告

**实现**：
- 使用 `try-except` 块包裹文件操作
- 使用 `logging` 模块记录警告和错误
- 在脚本结束时显示统计信息（成功/失败/跳过）

## Risks / Trade-offs

### Risk 1: 图片引用格式变化导致解析失败

**风险描述**：如果博客文章中的图片引用格式发生变化（如不同的 alt 文本或路径格式），解析可能失败。

**缓解措施**：
- 使用灵活的正则表达式，支持多种 alt 文本变体
- 在无法解析时记录详细警告，提示用户检查格式
- 在文档中说明期望的图片引用格式
- 未来可以考虑添加配置文件支持自定义模式

### Risk 2: 性能问题（大量 Markdown 文件）

**风险描述**：如果博客文章数量很大（数百篇），递归扫描和解析可能影响性能。

**缓解措施**：
- 使用生成器（yield）惰性处理文件，避免一次性加载所有内容到内存
- 关键字检查在图片引用解析之前进行，快速过滤无关文件
- 未来可以考虑添加缓存机制（如记录已处理的文件及其修改时间）

### Risk 3: 相对路径解析的歧义性

**风险描述**：不同的相对路径格式（`../`、`./`、绝对路径等）可能导致解析错误。

**缓解措施**：
- 实现健壮的路径标准化逻辑
- 测试多种路径格式，确保覆盖常见情况
- 在解析失败时提供清晰的错误信息

### Risk 4: 域名硬编码

**风险描述**：硬编码 `https://invest.zdyi.com` 可能导致在其他环境下失效。

**缓解措施**：
- 当前阶段接受硬编码，简化实现
- 在文档中说明域名配置
- 未来可以通过 `--base-url` 命令行参数支持自定义域名
