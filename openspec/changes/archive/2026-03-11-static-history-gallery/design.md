## Context

当前项目是一个基于 GitHub Pages 的个人网站，使用静态站点生成器。`docs/attachments/` 目录下存储了按年份组织（2022-2026）的子目录，每个日期目录包含一张交易截图（`1.png`）。

目前这些图片仅存储在文件系统中，没有可视化的历史回顾页面。需要一个独立的 Python 工具来：
- 扫描现有的目录结构
- 提取并整理图片
- 生成可浏览的静态 HTML 页面

**约束条件**：
- 输出必须是纯静态 HTML，无需后端服务
- 页面应独立于现有网站架构，便于维护
- 工具应易于在本地或 CI/CD 中运行
- 必须保持图片的原始路径（`docs/attachments/`），不复制文件

## Goals / Non-Goals

**Goals:**
- 创建一个 Python 脚本，能够自动扫描 `docs/attachments/` 目录并生成瀑布流图片展示页面
- 生成的 HTML 页面应具有响应式设计，适配桌面和移动设备
- 支持按时间倒序展示交易历史
- 提供友好的用户体验（图片点击放大、日期标签清晰）
- 工具应具有容错性，能够处理缺失的目录或文件

**Non-Goals:**
- 不实现图片上传或管理功能
- 不集成到现有网站的构建流程（仅提供独立工具）
- 不实现用户认证或权限控制
- 不创建数据库或动态后端
- 不修改原始图片文件

## Decisions

### 1. 技术栈选择：Python 标准库 + Jinja2

**决策**：使用 Python 3.8+ 标准库进行文件系统操作，使用 Jinja2 模板引擎生成 HTML。

**理由**：
- **Python**：易于编写和维护，强大的文件系统操作能力，跨平台兼容性好
- **Jinja2**：成熟稳定的模板引擎，语法清晰，便于维护 HTML 结构
- **为什么不使用其他方案**：
  - Bash 脚本：HTML 生成和维护困难，缺乏结构化数据处理能力
  - Node.js：需要额外的运行时环境，增加依赖复杂度
  - 现有网站生成器：可能过度集成，增加耦合度

### 2. 目录结构设计

**决策**：在项目根目录创建 `history/` 目录，包含以下结构：
```
history/
├── generate.py          # 主生成脚本
├── template.html        # Jinja2 模板文件
└── static/
    └── index.html      # 生成的输出文件（git 可选）
```

**理由**：
- 独立目录便于维护和版本控制
- 模板与代码分离，便于定制样式
- 生成脚本与输出文件在同一目录，使用直观

**替代方案考虑**：
- 将生成器放在 `scripts/` 目录：增加项目复杂度，不利于独立使用
- 将模板放在 `docs/` 目录：混淆内容和工具的边界

### 3. 日期解析策略

**决策**：使用多种日期格式解析策略，按优先级尝试：
1. ISO 8601 格式（YYYY-MM-DD）
2. 常见的日期格式（YYYY/MM/DD、YYYY.MM.DD）
3. 如果无法解析，使用目录名作为标签，并将其放置在列表末尾

**理由**：
- 容错性强，适应不同的目录命名习惯
- 确保即使日期格式不标准，图片仍能被展示
- 按时间排序对用户体验至关重要

### 4. 瀑布流布局实现

**决策**：使用 CSS Grid + `column-count` 属性实现纯 CSS 瀑布流布局，不依赖 JavaScript。

**理由**：
- **纯 CSS 方案**：
  - 无需 JavaScript，页面加载更快
  - 静态页面更安全，无客户端脚本执行风险
  - 更易于维护和定制
- **为什么不使用 JavaScript 库**：
  - 增加复杂度和加载时间
  - 对于静态内容，JS 方案优势不明显
- **CSS Grid 优势**：
  - 原生支持，性能优异
  - 响应式设计简单（使用 `@media` 查询）
  - 浏览器兼容性好

**实现细节**：
- 使用 CSS Multi-column 布局（`column-count`）实现真正的瀑布流效果
- 图片容器使用 `break-inside: avoid` 防止被分割
- 移动端：1 列，平板：2-3 列，桌面：3-4 列

### 5. 图片路径处理

**决策**：生成 HTML 时使用相对路径 `../docs/attachments/<year>/<date>/1.png`

**理由**：
- 保持文件系统整洁，不复制图片
- 生成的页面可以直接在 GitHub Pages 上访问
- 相对路径确保页面在不同目录层级下都能正常工作

**注意事项**：
- 需要验证生成的相对路径在部署环境中正确
- 图片必须在 `docs/` 目录下才能被 GitHub Pages 正确提供

## Risks / Trade-offs

### Risk 1: 日期解析失败导致排序错误
**风险描述**：如果目录名不符合预期的日期格式，图片可能无法正确排序。

**缓解措施**：
- 实现多种日期格式解析策略
- 无法解析时使用目录名作为标签，并放置在列表末尾
- 在生成时输出警告信息，提示用户检查目录命名

### Risk 2: 大量图片导致页面加载缓慢
**风险描述**：如果图片数量很大（数百张），页面可能加载缓慢。

**缓解措施**：
- 使用 `<img loading="lazy">` 属性实现懒加载
- 在生成器中添加图片数量检查和警告
- 未来可以考虑实现分页或虚拟滚动（超出当前范围）

### Risk 3: 相对路径在部署环境中失效
**风险描述**：生成的相对路径可能在不同部署环境中失效。

**缓解措施**：
- 在生成完成后测试生成的 HTML 文件
- 在文档中说明预期的目录结构
- 提供配置选项让用户自定义路径前缀（未来增强）

### Risk 4: 目录结构变化导致生成器失效
**风险描述**：如果 `docs/attachments/` 目录结构发生变化，生成器可能无法正常工作。

**缓解措施**：
- 添加详细的错误处理和日志输出
- 在文档中说明预期的目录结构
- 提供命令行参数允许自定义输入目录

## Migration Plan

### 部署步骤

1. **创建 `history/` 目录及生成脚本**
   - 编写 `generate.py` 主脚本
   - 创建 `template.html` Jinja2 模板文件
   - 添加 `requirements.txt`（如需额外依赖）

2. **本地测试**
   - 运行 `python history/generate.py`
   - 验证生成的 `history/static/index.html` 在浏览器中正确显示
   - 检查所有图片链接有效

3. **提交到版本控制**
   - 将 `history/` 目录添加到 git
   - 提交代码推送到 GitHub

4. **集成到 CI/CD（可选）**
   - 在 `gh-deploy.sh` 中添加生成脚本调用
   - 确保每次部署前重新生成 `history/static/index.html`

### 回滚策略

如果生成的页面出现问题：
- 直接删除 `history/static/index.html` 即可回滚
- 或从 git 历史中恢复之前的版本
- 不影响 `docs/attachments/` 中的原始图片

## Open Questions

1. **是否需要支持自定义输出路径？**
   - 当前设计固定输出到 `history/static/index.html`
   - 如果用户希望自定义输出位置，可以添加命令行参数

2. **是否需要支持自定义模板？**
   - 当前设计使用固定的 `template.html`
   - 如果用户希望定制样式，可以通过修改模板文件实现

3. **是否需要添加图片元数据支持？**
   - 当前仅从目录名提取日期信息
   - 如果需要更多元数据（如标题、描述），可以考虑添加配置文件（如 YAML）

4. **是否需要添加图片缩略图生成？**
   - 当前直接显示原图
   - 如果图片较大，可以考虑生成缩略图以提升性能
